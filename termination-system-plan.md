# SlimCast — Universal Termination System Plan

**Status:** PLAN — approved-direction, not yet implemented. No code changed as of this writing.
**Author:** termination audit (8-agent fleet) + design synthesis, 2026-06-28.
**Scope:** every watchdog / reaper / self-destruct / teardown path across **both** the VPS hub and the GPU backend (and the legacy all-in-one pod), made **lightweight, failproof, provider-agnostic, and reconnection-safe.**

> This document is the single source of truth for the termination redesign. It captures (a) the orphaned-Hetzner-VPS incident and its root cause, (b) an exhaustive inventory of every current termination path with file\:line citations, (c) every catchall / universality / reconnection gap found, (d) the locked design decisions, and (e) a concrete phased implementation plan with test strategy and acceptance criteria.

---

## 0. The ask (verbatim intent)

> "I want a **lightweight termination system that is failproof**, not a heavy system termination system that checks for individual logics — something simple that would be a **catchall** while still **allowing for stream reconnection** and **won't buckle under pressure of spotty connection**."
>
> "Please make these self-termination functions **universal** — I plan on using **more than Hetzner for VPS and more than Vast for GPUs**."
>
> "Could we use a **timer on a dead-man's switch** to allow for broadcast reconnection?"

The current system is the opposite of that ask: it is **heavy** (6 reaper sub-loops + an inline sweep + per-role self-destruct conditions), **provider-coupled** (hard-wired to Vast and Hetzner in a dozen places), and uses **one 150 s threshold that doubles as both reconnect-tolerance and orphan-reaping** — so it cannot be tuned for one without breaking the other.

---

## 1. The incident

A Hetzner hub VPS **did not self-delete** and had to be destroyed manually. A live API check during this audit showed **0 Hetzner servers running**, so the one orphan was the only one and nothing is billing now — but the structural hole that produced it is still open.

## 2. Root cause (confidence: **medium**)

> The card-less Hetzner hub (RELAY_ROLE=vps) has NO independent self-destruct and NO absolute time cap, so its survival is 100% dependent on the Vercel control plane reaching it AND a correct teardown decision being made — and both can fail silently. Grounded specifics: (1) relay/agent.py main_vps (lines 468-571, docstring 474-476) NEVER calls /api/agent/terminate and carries no provider API key, so the box can never kill itself; its only shutdown is a cloud-pushed command:stop reason=scale_to_zero (agent.py:562-567), which merely sys.exit(0)s the process and does NOT destroy the Hetzner VM (web-side teardownHub does). (2) vps_hubs has NO max_session_at / lease column (migration 20260628000002) — the only heartbeat-independent TTL in the schema lives solely on gpu_instances (max_session_at, security_hardening.sql:85). (3) The web-side Clock B has only three kill conditions (reap/route.ts:185-191): spawn_timeout, stale_hub (last_seen>150s), and scale_to_zero (session_count<=0 AND empty>10min). A hub that keeps heartbeating with a wedged session_count>=1 — caused by a lost detach_from_hub (it is a blind decrement, never reconciled against the real count of gpu_instances where vps_hub_id=id; and teardownInstance deletes the row at pod-teardown.ts:35 then calls detach as a separate un-try/catched op at line 56, or a CASCADE/account-deletion delete skips detach entirely) — matches NONE of those branches and bills forever. (4) Even in the simpler "agent died / network-partitioned but box still rented" case, every timely path (scale_to_zero, Clock A, handleVpsFailed) is agent-dependent and dead, leaving the DAILY cron stale_hub as the SOLE backstop — and web/vercel.json schedules reap once/day ("0 0 * * *", Vercel Hobby cap), so an orphan bills up to ~24 hourly-rounded Hetzner hours, which is why manual destruction beat the reaper. The two candidate triggers (wedged-refcount-immortal vs dead-heartbeat-waiting-for-daily-cron) cannot be separated without the incident logs, but the structural root cause is identical and certain: no box-local dead-man's switch + no absolute TTL on vps_hubs + the only agent-independent reaper runs daily.

**In one sentence:** the card-less Hetzner hub has **no box-local dead-man's switch and no absolute TTL**, so its survival depends 100% on the Vercel control plane both *reaching* it and *deciding correctly* to kill it — and the only agent-independent backstop (`/api/cron/reap`) runs **once per day**.

Two concrete trigger paths, same structural cause:

1. **Wedged refcount → immortal hub.** Hub liveness is a `session_count` that is a *blind decrement* (`detach_from_hub`), never reconciled against the real number of attached tenants. A single lost detach — an un-try/caught detach *after* the row is already deleted (`web/lib/pod-teardown.ts:56`), a stuck OBS flag, or a CASCADE delete — strands it `> 0` permanently. A hub at `session_count ≥ 1` matches **none** of the reaper's three kill branches, and `vps_hubs` has **no `max_session_at`/lease column**. It bills hourly, forever.
2. **Dead agent → only the daily cron catches it.** If the agent dies or the box is network-partitioned, every timely path (`scale_to_zero`, Clock A, `handleVpsFailed`) is agent-dependent and dead. The sole backstop is `stale_hub` in `/api/cron/reap`, scheduled `"0 0 * * *"` (Vercel Hobby cap) — so an orphan bills up to ~24 hourly-rounded Hetzner hours before reaping.

- HETZNER HUB, web never sends scale_to_zero: main_vps (agent.py:468-571) has no max_session/idle/heartbeat-fail self-destruct. A mis-tracked gpu_node (the documented racers-jsonb race class) is orphaned with no relay backstop.

- ALL-IN-ONE, terminate POST lost: OBS disconnects, grace timer fires (agent.py:789-792), POST /api/agent/terminate fails (transient 5xx). _do() neither retries nor exits; prev_obs_connected is already False so the elif never re-arms. **Provider coupling (breaks on a new provider):**
- Readiness self-report reads Vast-injected env directly: VAST_UDP_PORT_8890 / VAST_TCP_PORT_1935 / VAST_CONTAINERLABEL (agent.py:93-99) and VAST_TCP_PORT_8899 with a RUNPOD_TCP_PORT_8899 fallback (agent.py:615). A new VPS/GPU provider that does not inject VAST_*: PUBLIC_IPADDR falls back to ipify (_get_external_ip), but srt_port/rtmp_port/bridge_port become None and provider_id becomes '' — the cloud cannot match the pod in the racers jsonb or build its public URL. This is readiness, not teardown, but it is the relay's main provider coupling and would need new code per provider.
- The relay has NO provider teardown coupling at all — the flip side of provider-agnosticism. There is zero Vast/Hetzner API-key usage for destroy anywhere in agent.py/supervisor.py/budget.py. Adding a provider needs zero new relay TERMINATION code, but it also means the relay can NEVER independently kill a new provider's resource; 100% of actual destruction is web-side teardownInstance/teardownHub, which ARE provider-specific (lib/providers/*). So 'add a provider with zero termination code' is true only for the relay; the web side still needs a provider.destroy() + reaper reconciliation entry per provider.
- Self-destruct is fundamentally web-dependent: the only resource-removing actions the relay can take are HTTP POSTs to /api/agent/terminate (all-in-one only) or waiting for the status route to call teardown. The vps and gpu roles never POST terminate at all. So the universal/failproof time-based gate the user wants does NOT exist on the relay for the two newest roles — they are pure web-delegated.

**Reconnection risks on this surface:**
- DISCONNECT_GRACE_S=20s (agent.py:113) is the all-in-one OBS-reconnect window. A full SRT drop (network blip beyond SRT's tolerance, cellular handoff, OBS restart) clears /tmp/obs_connected via hook.sh runOnNotReady and starts the 20s clock; an OBS that reconnects at >20s finds the pod already terminating and must re-provision. Tight for spotty/mobile uplinks. Note SRT's own 5000ms latency buffer absorbs packet loss, so only a true publish-drop (not jitter) clears the flag — but 20s of true downtime kills the pod.
- All-in-one missed-heartbeat stop at ~60s (agent.py:950-956) keys on CONTROL-PLANE (Vercel) reachability, not OBS health: a 60s Vercel/network blip stops the user's healthy stream. Worse, it does NOT auto-resume — after stop_all the throttle-aware apply (agent.py:921-929) is gated on sig=(config_hash,tier), which is unchanged, so outputs stay dead even after heartbeats recover, until config/tier/OBS state changes. A transient control-plane jitter can therefore silently end a stream that OBS never dropped.
- GPU backend missed-heartbeat stop at ~60s (agent.py:692-696) similarly keys on control-plane reachability; a brief hub<->Vercel or GPU<->Vercel blip stops transcode. It does set last_hash=None so it re-applies when config returns (unlike all-in-one), so the false-positive is self-healing here — but the ~60s stop still interrupts the stream on jitter.
- VPS hub per-tenant schedule_stop uses Supervisor.grace_seconds=20s (supervisor.py:1002) — same tight 20s window per tenant on OBS drop. A tenant's spotty uplink that reconnects after 20s loses its pipeline (must re-trigger via OBS reconnect/hub-config re-apply).
- HEARTBEAT_FAIL_LIMIT (6) and the grace values are env-tunable, but the defaults bias toward stopping the stream on ~60s control-plane loss and killing the pod on ~20s OBS loss — both can false-positive under the spotty-connection conditions the user explicitly wants protected. None of these is a per-provider check, but all four are condition-based rather than the single simple time gate the user prefers.

- NEW PROVIDER ADDED TO ACTIVE_* ARRAYS BUT NOT TO THE RESOLVER MAPS: getProvider/getVpsProvider throw 'Unknown provider' (index.ts:40-46/64-71); teardownInstance (try/catch, lines 59-65/94-98) and teardownHub (try/catch, line 195) SWALLOW the throw best-effort → the box is silently never destroyed and leaks. 'Adding a provider needs zero termination code' is only true if you also register it in PROVIDERS / VPS_PROVIDERS, otherwise it leaks silently.
- CONTROL-PLANE OUTAGE: both the Vast pod and the Hetzner hub depend on Vercel to issue teardown — neither hits its provider's destroy API directly from the box. If Vercel/cron is down or misconfigured, every running hub and GPU backend orphans with nothing local to kill it. The hub is worst because it has no self-destruct and no TTL at all.
- teardownHub Vercel-function timeout mid-loop: it nulls each tenant's vps_hub_id and status='error' (line 156) and deletes relay_nodes rows one-by-one (line 186) BEFORE destroying the hub box (line 191). If the serverless function is killed partway, some gpu-backends have their relay_nodes row deleted but the box still alive and now unreferenced → caught only by the daily orphan-reconcile.

**Provider coupling (breaks on a new provider):**
- getProvider() falls back to vastProvider on a null/empty provider value (web/lib/providers/index.ts:38). Any future GPU provider row that ever writes a NULL/empty provider would be mis-routed to Vast.destroy() — a no-op against the wrong API → silent leak. This implicit Vast default is provider-coupled.
- Resolver maps are hand-maintained: PROVIDERS = {vast, runpod} (index.ts:24-27) and VPS_PROVIDERS = {hetzner} (index.ts:56-58). A new provider needs adding here AND to ACTIVE_PROVIDERS / ACTIVE_BACKEND_PROVIDERS / ACTIVE_VPS_PROVIDERS, or teardown throws and the reaper's orphan reconcile never lists it. So termination requires per-provider registry edits — not literally zero code.
- relay/agent.py:615 reads the GPU bridge port from hard-coded env names VAST_TCP_PORT_8899 OR RUNPOD_TCP_PORT_8899. A new GPU provider's port-mapping env var is not read → the GPU backend can't report its bridge port (a pairing failure, but it compounds orphan risk: an unpaired-but-alive box relies on the never_paired sweep).
- Hub data-plane self-shutdown is keyed to a cloud command only (relay/agent.py:562 scale_to_zero). There is no provider-injected self-destruct credential on the hub (contrast the Vast pod's /root/.vast_api_key noted in project docs). Every VPS provider added will inherit this 'cannot self-destruct' property unless a generic box-side TTL kill is built.
- Orphan-reconcile relies on each provider.listInstances() honoring a name/label convention (Vast: name startsWith 'slimcast-'; Hetzner: label managed-by=slimcast + name 'slimcast-hub-'). A new provider whose listInstances doesn't reproduce this exact naming/labeling is invisible to the only DB-independent leak catcher.

**Reconnection risks on this surface:**
- teardownHub stale_hub (>150s no heartbeat) destroys the hub box UNCONDITIONALLY even with live tenants attached — it errors all tenants (status='error', vps_hub_id=null, pod-teardown.ts:154-157) and tears down their GPU backends. A ~150s Vercel<->hub control-path blip on an ACTIVE multi-tenant hub would nuke the whole box plus every tenant's stream. Mitigated only by the daily cron cadence (low probability of coinciding), but the threshold itself is aggressive for a shared box with no 'is the data plane actually still flowing?' check.
- gpu-backend mid-stream re-race at stale 150s (reap/route.ts:280,295): a 150s gap in the GPU's CONTROL heartbeat triggers reraceGpuBackend — destroying a possibly-healthy transcoding GPU and racing a new one. The actual hub<->GPU data bridge (mpegts-over-TLS) is independent of this control heartbeat, so a control-plane jitter could churn a GPU that is still delivering frames. Daily cadence limits exposure.
- Pod/hub stale=150s with 10s heartbeats = 15 consecutive missed beats before kill, and idle grace is 5m — these are reasonably reconnection-tolerant (good). The genuinely reconnection-safe path is the relay pod's DISCONNECT_GRACE_S=20s timer with cancel_terminate on OBS reconnect (relay/agent.py:786-798) — but that exists ONLY for the legacy pod, not for hub tenants, whose reconnect tolerance is the coarser 5m server-side idle grace.
- Because billing-driven kill (credits<=0) is currently gated OFF (SLIMCAST_BILLING_ACTIVE=false), the live kill triggers are idle/max_session/stale — none of which false-positive on a brief OBS drop. The risk is concentrated on the stale-heartbeat hub destroy above, not on user-facing reconnects today.

**Orphan scenarios on this surface:**
- HUB STUCK-REFCOUNT (prime incident suspect): teardownInstance deletes the gpu_instances row (pod-teardown.ts:35-42) and THEN calls detach_from_hub as a SEPARATE op (line 56) — not one transaction. If the serverless invocation dies/timeouts between the delete and the detach, session_count is never decremented. The hub keeps heartbeating, empty_since stays null, session_count stays >0 forever. Neither handleVpsStatus scale-to-zero (needs session_count<=0) nor reaper stale_hub (needs no-heartbeat) nor reaper scale_to_zero (same condition) ever fires. The hub has NO max_session_at. Result: a healthy hub box bills forever until manual destroy. The reaper orphan-reconcile cannot help — it only catches boxes with NO vps_hubs row, and this hub HAS a row.
- GPU BACKEND with a live but wedged parent: handleGpuStatus never kills and returns credits_seconds=999999, and main_gpu() never self-exits. If the parent gpu_instances tenant row is somehow updated to look 'running' (e.g. handleVpsStatus line 118-120 promotes any vps_hub_id tenant in status='provisioning' to running on every hub beat) while no OBS is actually connected, the GPU backend keeps billing. It relies entirely on Clock A idle (needs the hub to report streaming=false for that tenant) — if the hub mis-reports or the ingest_key match fails (status/route.ts:153), idle_since may not advance correctly.
- Raw gpu_instances delete bypassing detach/gpuNodes-capture: releaseClaim in provision (provision/route.ts:138-140) does a bare gpu_instances.delete().eq('user_id') with NO detach_from_hub and NO gpu_backend capture. Not reachable in today's hub flow (provision returns before releaseClaim once a hub is attached), but it is a latent landmine: any future raw delete of a hub-attached row leaks both the hub refcount AND any gpu_backend box (FK CASCADE drops relay_nodes without calling provider.destroy).
- Hub heartbeat omits all streams: if a hub relay (or hub-config) returns an empty stream set, every tenant goes idle and is torn down after 5 min, which does eventually drop session_count to 0 → scale-to-zero in another 10 min. Converges, but ~15 min of full-hub idle billing before the box dies, and only if the hub keeps heartbeating.
- No-beat hub between cron runs: if a hub stops heartbeating (box/agent dead) it is caught ONLY by reaper stale_hub>150s, but the cron runs daily on Vercel Hobby (cron/reap comment lines 8-14). A dead-but-still-rented hub can bill up to ~24h before the daily reaper destroys it.

**Provider coupling (breaks on a new provider):**
- teardownInstance and teardownHub resolve the destroy target by STORED provider name via getProvider()/getVpsProvider() (pod-teardown.ts:61,95,169,193) — these are provider-neutral, but each new provider MUST be registered in the resolver maps (providers/index.ts:24-27 PROVIDERS and :56-58 VPS_PROVIDERS) AND in an ACTIVE_* array. getProvider/getVpsProvider THROW on an unknown name (providers/index.ts:40-47,64-71) — so a forgotten registry entry leaks the box LOUDLY (error logged, box survives) rather than mis-routing. Not 'zero code' per the user's goal #2, but no per-provider termination LOGIC — just a 2-line registry entry.
- The heartbeat ROLE dispatch is provider-agnostic in name only: handleVpsStatus/handleGpuStatus are selected by key label ('vps'/'gpu' in authenticateNode, agent-auth.ts:80-99), not by provider — good. But the GPU backend's bridge_port env read in agent.py:615 hard-codes VAST_TCP_PORT_8899 / RUNPOD_TCP_PORT_8899 — a third GPU provider's port-injection env name is NOT read, so its backend would report bridge_port=None.
- handleGpuStatus returning credits_seconds=999999 unconditionally (status/route.ts:87) means NO provider's GPU backend can ever self-destruct via heartbeat — the only kill is cloud-side. Adding a GPU provider does not add any self-destruct safety; it inherits the same total dependence on Clock A / teardownHub / daily reaper.
- The reaper orphan-reconcile (the true catchall) keys on box NAME prefix slimcast- / slimcast-hub- and on provider.listInstances() (cron/reap:146-167,215-232). A new provider must implement listInstances() with the right label/name filter and be added to ACTIVE_PROVIDERS / ACTIVE_BACKEND_PROVIDERS / ACTIVE_VPS_PROVIDERS or its orphans are invisible — provider coupling in the safety net, not in this heartbeat surface but it is what backstops this surface.

**Reconnection risks on this surface:**
- Legacy pod 20s OBS-disconnect kill is aggressive: the relay's own watchdog calls /api/agent/terminate 20s after OBS drops (DISCONNECT_GRACE_S, agent.py:113,786-796), which runs teardownInstance and DESTROYS the pod. A 25-30s OBS blip (network hiccup, OBS restart) destroys the GPU pod — the user must re-provision (~45s). The heartbeat-side 5-min idle grace is far more forgiving but the relay's 20s timer beats it every time. This is the tightest reconnection gate in the system.
- sweepStalePods + reaper stale_heartbeat=150s destroy a LIVE pod whose CONTROL-PLANE heartbeats (pod->Vercel) fail >150s even when the DATA plane (OBS->pod->platforms) is healthy. Spotty pod->Vercel connectivity false-positive-kills a working stream. Compounded: the relay stops OUTPUTS at HEARTBEAT_FAIL_LIMIT=6 (~60s) of missed beats (agent.py:953-955), so a 60-150s control-plane blip darkens an otherwise-fine stream, and >150s destroys the pod.
- Hub-tenant idle is keyed on the hub matching the tenant by ingest_key (status/route.ts:153). If a hub heartbeat transiently drops a tenant from its reported stream set (relay restart, hub-config blip), that tenant is treated as streaming=false → idle_since starts. A real OBS reconnect within 5 min clears it (safe), but repeated flapping near the 5-min boundary could trip Clock A idle teardown of a user who is actually mid-stream.
- handleVpsStatus promotes ANY status='provisioning' tenant to running/ready on every hub beat (status/route.ts:118-120). Combined with idle_since defaulting to nowIso when not streaming (line 155), a tenant whose OBS hasn't connected yet immediately starts the 5-min idle clock the moment its hub goes live — a slow OBS first-connect (>5 min after hub ready) would be idle-killed even though the user never got to reconnect.
- GPU backend stop-on-missed-heartbeat is NOT a reconnection kill (good): main_gpu only sup.stop_all() after 6 missed beats and keeps the box alive (agent.py:693-695), so a control-plane blip pauses transcode but does not destroy the GPU — it resumes on the next config. But because it never self-destructs, the inverse risk (orphan) dominates over the false-positive risk for the GPU role.

- NONE in the plugin — and that is by design. relay-api.cpp hard-codes only BASE_URL (relay-api.cpp:20) and hits provider-neutral endpoints (/api/gpu, /api/gpu/provision, /api/gpu/status). - Caveat (not coupling, but a universal blind spot): the plugin has no hub abstraction at all. So the inability to terminate a hub is UNIVERSAL across every VPS provider, present and future; it is not a Hetzner-specific gap.

**Reconnection risks on this surface:**
- Stream watchdog 90s (relay-dock.cpp:202-209): on a spotty/jittery uplink where OBS takes >90s to complete the FIRST SRT handshake to the pod, the watchdog stops OBS and destroys the pod — a false-positive kill at initial connect. It is single-shot and not re-armed mid-stream, so true mid-stream reconnects are NOT affected.
- onObsStreamingStopped → immediate destroyGpu (relay-dock.cpp:740) has NO grace period. A SHORT OBS disconnect is safe — OBS keeps the output active during its own internal auto-reconnect, so obs_frontend_streaming_active() stays true and nothing fires. But once OBS exhausts its reconnect retries on a LONG outage and fully stops the output, STREAMING_STOPPED fires and the pod (and gpu_backend) are destroyed instantly. There is no plugin-side window to let the user resume against the still-good pod — they must Go Live again and eat a full ~45s re-provision.
- Orphan auto-destroy (relay-dock.cpp:808) is keyed on obs_frontend_streaming_active(). During OBS's internal reconnect that flag stays TRUE, so this path does NOT false-positive. The extra 2-poll (~10s) debounce (m_orphanTicks>=2) further guards against racing the launch/resume window. LOW risk — this path is correctly conservative.
- Net: the plugin never kills on missed heartbeats (it has no heartbeat concept — it polls /api/gpu/status read-only). Reconnection tolerance is delegated entirely to OBS's own reconnect behavior; the only aggressive kill is the 90s initial-connect watchdog.

- CASCADE leak on account deletion: profiles ON DELETE CASCADE → gpu_instances ON DELETE CASCADE → relay_nodes. Deleting a user account drops all three DB rows but calls neither provider.destroy() (Vast pod / GPU backend) nor detach_from_hub() (Hetzner hub refcount). Every cloud box the user owned leaks; the shared hub's refcount is stranded high. No DB trigger or teardown hook fires on cascade.
- Racer-only box: a v2 race creates N boxes; only the winner's id is promoted to the top-level provider_id. If the racers-jsonb write loses a race or the function dies mid-fan-out, a live racer box exists whose id is in NO column a per-row (provider,provider_id) reaper reads. Today this is caught only by bespoke racer loops + the name-prefix orphan sweep (reap/route.ts:86-167) — not by any lease/TTL, so a lease-based universal reaper would still miss it.
- gpu_backend with wedged parent: a relay_nodes gpu_backend goes stale while its parent gpu_instances is stuck status='running' with a still-heartbeating hub. The reaper classifies parent as sessionLive (reap/route.ts:290) and RE-RACES a fresh GPU instead of destroying — potentially spawning replacement GPU boxes indefinitely off a zombie session, with no absolute cap because relay_nodes has no max_session_at.

**Provider coupling (breaks on a new provider):**
- Two separate destroy registries keyed on resource TYPE, not provider: getProvider() (GPU: vast, runpod) vs getVpsProvider() (VPS: hetzner) in web/lib/providers/index.ts. A generic lease reaper cannot do one getProvider(row.provider).destroy(row.provider_id) across all billable rows — it must branch on whether the row is a hub (→ getVpsProvider) or a GPU box (→ getProvider). The SCHEMA columns (provider, provider_id) are provider-agnostic; the RESOLVER selection is resource-type-coupled.
- vps_hubs.primary_ip_id (20260628000002_vps_hubs.sql:18) is a Hetzner-shaped side resource — a second billable IPv4 that survives server deletion and must be released separately (VpsProvider.destroy(id, {primaryIpId})). A future VPS provider with a different side-resource model (Vultr reserved IP, etc.) cannot be expressed by this single Hetzner-named column; the schema bakes in Hetzner's IP-leak shape.
- Orphan reconcile depends on naming conventions, not on schema columns: the reaper matches row-less boxes by name prefix 'slimcast-' for GPUs (reap/route.ts:156) and 'slimcast-hub-<region>-<hubId8>' for hubs (reap/route.ts:219-221). Any new provider's listInstances() must emit a name following these exact conventions or its orphans are invisible. This is a soft per-provider coupling outside the DB schema.
- The reaper has three hardcoded per-table sweeps (gpu_instances, relay_nodes, vps_hubs) with table-specific column sets and thresholds. Adding a new PROVIDER within an existing resource type needs only a registry entry (zero termination code) — that part meets the user's goal. But the schema offers no single unified billable-resource view, so adding a new RESOURCE TYPE (not provider) requires new reaper code. The schema is provider-agnostic but resource-type-fragmented.
- gpu_instances.provider defaults to 'runpod' (20260618000004_gpu_broker.sql:5) then semantically 'vast' — a stale default. getProvider() treats NULL/'' as legacy Vast but THROWS on unknown non-empty names (good, fail-loud), so an un-registered future provider name on a row would make the reaper throw rather than leak — but only if the row reaches getProvider; the lease scan itself has no provider validation.

**Reconnection risks on this surface:**
- STALE_S=150s on last_seen_at is the ONLY reconnection buffer for every table (reap/route.ts:17, pod-teardown.ts sweep, vps_hubs Clock B). With ~10s heartbeats that's ~15 missed beats. A 2.5-minute network blip during a live stream false-positives as 'stale' and tears down. The user explicitly does not want kills on spotty connections — 150s is thin, and it is duplicated as a magic number across reaper + inline sweep + the attach RPC liveness guard (20260628000005:28) rather than a single tunable lease length.
- relay_nodes mid-stream re-race fires at the same 150s stale (reap/route.ts:295) even when the GPU backend is merely network-blipped, not dead — it would destroy/replace a healthy GPU on a transient bridge hiccup, mid-stream, while the user is live.
- The hub-tenant skip (`if (inst.vps_hub_id) continue`, reap/route.ts:54 and pod-teardown.ts sweep) is the safest path for reconnection (a hub-mode session is never killed on its own missed heartbeats) — but it is the exact same line that creates the immortal-hub orphan. Tightening one direction loosens the other; the schema gives no independent lease to separate 'tolerate reconnect' from 'catch orphan'.
- idle_since is stamped the moment streaming flips false (a brief OBS disconnect), then a 300s grace applies. A reconnect within 5 min is safe — but the 5-min idle clock and the 150s stale clock can race: a stream that disconnects and the agent also stops heartbeating hits the 150s stale path first (kill) before the 300s idle grace can protect the reconnect.

**Orphan scenarios on this surface:**
- Wedged-but-heartbeating Hetzner hub: a tenant's OBS-connected flag file (/tmp/obs_connected.<key>) is stuck on (hook.sh disconnect didn't fire / stale file) → hub keeps reporting streaming=true → Clock A (status/route.ts:189) never idles it out → detach_from_hub never runs → session_count stuck >=1 → empty_since never set → scale-to-zero (status/route.ts:210) never fires → hub keeps heartbeating so reaper stale_hub never fires → reaper Clock B (reap/route.ts:176-201) has NO max-lifetime branch → the box bills FOREVER. Nothing kills it. This is the incident shape.
- Dead-but-running Hetzner box: network partition or agent crash while the server is still rented and billing. No heartbeat → Clock A, scale-to-zero, and (in hub-only deploys) the inline sweep are all dead → ONLY the daily cron stale_hub (reap/route.ts:187) catches it → up to ~24h of billing. Manual destroy was faster than the once-a-day cron — exactly the reported incident.
- GPU backend shielded by the orphan-prefix guard: a gpu_backend box (named slimcast-<userid8>) leaks — its relay_nodes row was CASCADE-dropped before teardownInstance captured it, or a re-race discarded its ids without a successful destroy. The user still has an active hub gpu_instances row, so knownUserPrefixes (reap/route.ts:139) contains <userid8> → the GPU orphan reconcile (reap/route.ts:159) SKIPS the box as 'mid-provision' → it bills until the user's session finally ends; if the user streams continuously, indefinitely.
- GPU backend re-raced forever: parent hub session is wedged-live (scenario 1) so sessionLive stays true; the reaper node sweep keeps hitting branch (b)/(b2) (reap/route.ts:295,311) and RE-RACES the GPU instead of destroying it → a GPU backend is perpetually kept/replaced, billing tied to the phantom tenant.
- Single-cron failure = total backstop loss: the daily /api/cron/reap is the SOLE independent killer for hubs and GPU backends (they have no relay self-destruct, no inline sweep). If that one cron silently stops — Vercel plan/cron-limit change, a deploy that drops the crons entry in web/vercel.json, or a CRON_SECRET mismatch returning 401 (reap/route.ts:34) — every hub and GPU backend loses its only catchall and accrues unbounded cost.
- Mid-race loser GPU never reaped: a racer box whose /ready never resolved and whose racers[] write lost a concurrency race → its provider_id is absent from knownPodIds, name is slimcast-<userid8>, prefix shielded if the user has any gpu_instances row → orphan reconcile skips it. (Same prefix-shield class as scenario 3.)

**Provider coupling (breaks on a new provider):**
- relay/agent.py:615 — GPU-backend bridge port is read ONLY from VAST_TCP_PORT_8899 or RUNPOD_TCP_PORT_8899. A new GPU provider's injected port-env name is unread → bridge_port=null → /ready reports a null bridge address. Provider-coupled at the data plane.
- relay/agent.py:596 (_post_ready_gpu) + VAST_INSTANCE_ID — the GPU self-reports a Vast-specific instance id; a new provider's self-id env isn't captured.
- web/app/api/cron/reap/route.ts:146 — gpuProviders is built from [...ACTIVE_PROVIDERS, ...ACTIVE_BACKEND_PROVIDERS]. A new GPU provider must be registered in (a) PROVIDERS map (getProvider), (b) ACTIVE_PROVIDERS and/or ACTIVE_BACKEND_PROVIDERS for the orphan sweep. Miss (b) and teardown works but the row-less orphan reconcile never lists the provider → forever leak. Three arrays to keep in sync.
- web/app/api/cron/reap/route.ts:215 — VPS orphan loop iterates ACTIVE_VPS_PROVIDERS; getVpsProvider resolves VPS_PROVIDERS. A new VPS provider must appear in both, AND name boxes slimcast-hub-<region>-<hubId8>, AND support a managed-by=slimcast queryable label (hetzner.ts:283 uses label_selector). A provider that can't label/name-filter its list has NO orphan backstop.
- Name-prefix matching is the orphan reconcile's only identity check: GPU = startsWith('slimcast-') + slice(8) user prefix; VPS = startsWith('slimcast-hub-') + last '-' segment. Any new provider that names instances differently is silently un-reconcilable.
- web/lib/providers/index.ts:38 getProvider default '?? vast' for null/empty provider, and vps-broker.ts:430 reraceGpuBackend uses `(node.provider) \|\| 'vast'` to route the dead-winner destroy. relay_nodes gpu_backend rows are INSERTed with provider:'' (vps-broker.ts:330) and only stamped on /ready. If a non-Vast racer won but node.provider is still '' when a destroy routes through the fallback, it calls Vast.destroy() against a non-Vast id → no-op → that box leaks. Latent, but it is a provider-coupled silent-fallback of exactly the kind index.ts:35-47 warns against.
- TWO separate registries (GpuProvider via getProvider vs VpsProvider via getVpsProvider) with DIFFERENT destroy signatures (VPS needs {primaryIpId}). Any universal sweeper must dispatch on resource kind; there is no single getProvider-for-anything entry point.

**Reconnection risks on this surface:**
- relay/agent.py:113 DISCONNECT_GRACE_S=20s — the all-in-one pod self-terminates 20s after OBS drops. A 25s uplink blip destroys the whole pod and forces a ~45s re-provision. Too aggressive for the stated 'allow brief reconnection' goal (the hub path's 300s Clock A is far safer; the legacy pod is the outlier).
- reap/route.ts:64 stale_heartbeat>150s and pod-teardown.ts:238 (inline, same 150s) do NOT check `streaming` — a pod that is healthily delivering to platforms but has a >150s control-plane heartbeat gap (clustered Vercel cold-starts / transient network) is DESTROYED mid-stream. Correct as a zombie-safety tradeoff, but a real false-positive-kill vector on spotty cloud connectivity.
- reap/route.ts:280-303 GPU node sweep: a healthy GPU backend with a >150s bridge/heartbeat blip while sessionLive triggers a destructive RE-RACE (kills the working GPU, spins a new one, brief transcode gap). False-positive churn rather than a leak, but it kills a working box.
- reap/route.ts:187 stale_hub>150s destroys a whole shared hub (all tenants errored, gpu_instances set status='error') on a 150s heartbeat gap — one transient control-plane outage tears down every tenant on that box. Blast radius = all N tenants, not one.
- relay/agent.py:693 HEARTBEAT_FAIL_LIMIT=6 (~60s): both the all-in-one pod (stops outputs) and GPU backend stop their pipelines after 60s of missed relay→cloud heartbeats. This is the relay→Vercel link (not OBS), so a 60s Vercel/network blip stops a live stream's outputs even while OBS is still connected. Recoverable (no destroy) but a user-visible interruption on a flaky control-plane link.

## 5. Catchall gaps — resources with NO independent backstop

- Hetzner hub (vps_hubs) with a wedged/non-zero session_count and a live heartbeat: matches NONE of Clock B's three branches (spawn_timeout / stale_hub / scale_to_zero) and vps_hubs has no max_session_at -> bills hourly forever. session_count is a blind decrement (detach_from_hub) never reconciled against count(*) gpu_instances where vps_hub_id=id, so a single lost detach (un-try/catched detach at pod-teardown.ts:56 after the row delete, a stuck OBS flag preventing idle, or a CASCADE delete) strands it >0 permanently. This is the most likely incident cause.
- GPU backend (relay_nodes role=gpu_backend): no relay self-destruct (main_gpu never terminates), no inline sweep (sweepStalePods skips it and is pod-gated), no max-lifetime column. Sole independent backstop is the DAILY cron node sweep — and if the parent session is wedged-live it is RE-RACED forever instead of destroyed.
- gpu-backend leaked after its relay_nodes row is gone (detach_from_hub throw in teardownInstance, or account-deletion profiles->gpu_instances->relay_nodes CASCADE with no provider.destroy): invisible to the node sweep; recoverable only by the daily name-prefix orphan reconcile, which is itself shielded by knownUserPrefixes whenever the user still has any gpu_instances row.
- Hetzner primary IPv4 (vps_hubs.primary_ip_id, auto_delete=false): NO primary_ips sweep exists anywhere in the reaper, directly contradicting the hetzner.ts:238 comment. A leaked IP bills ~EUR0.50/mo indefinitely.
- Racer boxes stuck in 'booting'/'racing' (not 'loser'/'failed') with a real provider_id in the racers jsonb: the racer-cleanup loop only destroys loser/failed states, so a stuck racer on an otherwise-healthy parent row is never swept by that loop.
- Any box whose DB row vanishes via the profiles->gpu_instances->relay_nodes CASCADE chain (account deletion or raw admin delete): no DB trigger or teardown hook fires, no provider.destroy, no detach_from_hub. Every cloud box leaks and the shared hub refcount is stranded high; only the name-prefix listInstances reconcile can recover it (a lease/TTL cannot — the row with the deadline is gone).
- Total cron loss: the daily /api/cron/reap is the ONLY independent killer for hubs and gpu-backends. A dropped vercel.json crons entry, a CRON_SECRET mismatch (401 at reap:34), or a Vercel plan change silently removes the sole catchall for both resources with no alarm.

## 6. Universality gaps — what breaks when adding a 2nd VPS / GPU provider

- Split, hand-maintained registries: PROVIDERS={vast,runpod} (index.ts:24-27) for GPUs vs VPS_PROVIDERS={hetzner} (index.ts:56-58) for VPS, PLUS three separate ACTIVE_* arrays (ACTIVE_PROVIDERS, ACTIVE_BACKEND_PROVIDERS, ACTIVE_VPS_PROVIDERS). A new provider must be added to the resolver map AND the correct ACTIVE_* array. Miss the map -> getProvider/getVpsProvider THROWS and the throw is swallowed by teardown try/catch -> the DB row is deleted and the box leaks silently. Miss the array -> the orphan-reconcile listInstances loop never lists the provider -> row-less boxes leak forever.
- Hard-coded box-name prefixes are the ONLY identity check in the row-less orphan reconcile: GPU = startsWith('slimcast-') + slice(8) user prefix (reap:156), VPS = startsWith('slimcast-hub-') + hubId8 name-tail parse (reap:219-221). A new provider whose create() ignores the passed name or labels boxes differently is invisible to the only DB-independent catchall.
- Implicit Vast fallbacks: getProvider() returns vastProvider for null/empty provider (index.ts:38); vps-broker.ts:430 reraceGpuBackend routes a dead box via (node.provider)\|\|'vast'; ready/route.ts:179 defaults winnerProvider to 'vast'. relay_nodes gpu_backend rows are INSERTed with provider:'' and only stamped on /ready — so a non-Vast racer that won but never stamped provider routes its destroy to Vast.destroy() against a foreign id -> no-op -> leak.
- Bridge-port data-plane coupling: relay/agent.py:615 reads the GPU bridge port ONLY from VAST_TCP_PORT_8899 / RUNPOD_TCP_PORT_8899. A third GPU provider's port-injection env name is unread -> bridge_port=null -> handleGpuReady 400 (ready:14) -> the box never wins/pairs -> orphaned until the never_paired sweep. Same for VAST_INSTANCE_ID self-id capture.
- Side-billable resources are Hetzner-shaped and unmodeled for others: vps_hubs.primary_ip_id is a single Hetzner-specific column; VpsProvider.destroy takes {primaryIpId}. A new VPS provider with a different detached-resource model (Vultr reserved IP, floating IPs, volumes, load balancers) has no schema slot and no release path, and there is NO generic detached-resource/primary-IP sweep at all.
- Two registries with DIFFERENT destroy signatures (VPS needs primaryIpId) mean no single universal sweeper is possible without a resource-kind discriminator on each billable row to pick getProvider vs getVpsProvider. The schema columns (provider, provider_id, last_seen_at) ARE uniform across all three tables, but the resolver selection is resource-type-coupled.
- Orphan reconcile requires each provider's listInstances() to support a queryable managed-by label/name filter (hetzner.ts uses label_selector). A provider that cannot list-filter its own boxes by an ownership tag has NO row-less orphan backstop whatsoever.

## 7. Reconnection risks — places that kill a healthy stream

- Legacy pod 20s DISCONNECT_GRACE_S (agent.py:113, 786-796) is the TIGHTEST gate in the system: a 25-30s OBS blip (cellular handoff, OBS restart) destroys the whole pod and forces a ~45s re-provision. SRT's 5000ms buffer absorbs jitter, so only a true publish-drop clears the flag — but 20s of true downtime kills.
- OBS plugin 90s relay-reach watchdog (relay-dock.cpp:202-209): a slow/jittery uplink that needs >90s to complete the FIRST SRT handshake gets its pod destroyed. Initial-connect only (not re-armed mid-stream), so true mid-stream reconnects are safe.
- 150s stale_heartbeat is duplicated across reap (route.ts:64), inline sweepStalePods (pod-teardown.ts:238), and vps_hubs Clock B (reap:187) and keys on CONTROL-PLANE reachability (box->Vercel), NOT the data plane. A >150s Vercel/network blip on a perfectly healthy stream destroys the box. For the shared hub this is catastrophic: teardownHub errors ALL tenants (status='error', vps_hub_id=null) on one 150s control-plane hiccup — blast radius is the whole multi-tenant box.
- gpu-backend mid-stream re-race at stale 150s (reap:295): a healthy transcoding GPU whose control heartbeat blips >150s (while the actual VPS<->GPU mpegts-over-TLS bridge is fine) is destroyed and replaced, churning a working box and interrupting transcode. A coincident hub-heartbeat blip can flip it from re-race(b) to destroy(a) -> GPU killed with NO replacement mid-stream.
- ~60s HEARTBEAT_FAIL_LIMIT=6 (agent.py:693): a 60s Vercel/network blip makes the all-in-one pod AND every gpu-backend stop outputs even though OBS and the hubs are healthy. The all-in-one path does NOT auto-resume (apply sig unchanged) -> a transient control-plane jitter silently ends a stream OBS never dropped; gpu-backend self-heals (last_hash=None).
- Hub tenant idle false-start: handleVpsStatus promotes any status='provisioning' tenant to running on every hub beat (route.ts:118-120) and idle_since defaults to now when not streaming (line 155) -> a slow OBS first-connect (>5min after hub ready) is idle-killed before the user ever connected. Repeated flapping near the 5-min boundary can also trip Clock A on a mid-stream user.
- A SINGLE 150s threshold doubles as both reconnect-tolerance and orphan-reap window: widening it to protect spotty connections directly delays orphan reaping, and tightening it to reap faster false-positive-kills healthy streams. There is no independent lease that separates 'tolerate reconnect' from 'catch orphan'.

## 8. Locked design decisions (interview, 2026-06-28)

| # | Decision | Choice | Notes |
|---|---|---|---|
| 1 | **Reconnect grace window** | **~3 minutes**, uniform | Replaces the legacy 20 s OBS-disconnect kill. Covers OBS restarts, cellular handoffs, short ISP drops. This is the *tenant source-idle* timer, not the box-liveness timer. |
| 2 | **Hub topology** | **Multi-tenant** (margin play) | Approved with one hard rule: **never store a refcount — derive emptiness from live leases** (see §9.3). This makes the orphan bug structurally impossible while keeping the margin upside. |
| 3 | **Catchall location** | **Control-plane sweeper only** | Boxes stay **key-free** (no provider token on the card-less hub). Box-local OS self-kill is explicitly *out of scope*. |
| 4 | **Sweeper cadence** | **Heartbeat-driven** | Every live relay's heartbeat opportunistically sweeps stale rows — no daily-cron dependency. Low-freq cron kept only as a floor for the all-idle case. |
| 5 | **Absolute max-lifetime** | **Per-tenant 12 h, confirm-to-extend** | Mirrors architecture #12. The **shared box has no hard cap** (warm-pool-friendly) — it dies only on its own lease expiry or derived-emptiness. |
| 6 | **Provider-specific logic** | **Unify (Phase 2)** | Single registry + generic managed-by label + generic detached-resource release; remove implicit `'vast'` fallbacks. |

### 8.1 Why multi-tenant is safe here (the reasoning requested)

Single-tenant is the *simplest* failproof catchall — teardown is just "destroy the box," with no "is anyone still using it?" question to get wrong (that question, answered wrong, is exactly what orphaned the hub). But multi-tenant is the right call for margins, and it does **not** have to reintroduce the bug:

- **The hub does no transcode** — it is a bandwidth mux + TLS endpoint (SRT ingest, passthrough delivery, GPU bridge). CPU per tenant is modest; the binding limit is bandwidth, and Hetzner gives **20 TB included per box** plus a **monthly cost cap** (~€4.5 for a box running 24/7). A small **warm pool** of always-on multi-tenant hubs amortized over many streams is far cheaper than one hourly box per concurrent stream — and doubles as the deferred sub-10 s spin-up warm pool. At 1,000 concurrent streams, single-tenant ≈ €4k+/mo of VPS; packed 20-to-a-box ≈ €200/mo.
- **The orphan bug is eliminated by deriving emptiness from leases** instead of a stored counter (§9.3). A `SELECT count(*) … WHERE renew_deadline > now()` is reconciled by construction every time it runs; a lost detach can't strand it because a dead tenant simply stops renewing.
- **Residual multi-tenant costs are not about termination** and are all manageable: noisy-neighbor bandwidth contention, shared-box blast radius (one box dies → its tenants reconnect via SRT + the 3-min window), and packing logic (which hub a new stream joins, max tenants/box).

## 9. Target architecture — one universal lease, two timers, one sweeper

The design collapses the 6 reaper sub-loops + inline sweep + per-role self-destruct into a **single time-gate model** that is provider-blind by construction.

### 9.1 The two timers (the dead-man's switch, done right)

The fatal flaw today is that **one 150 s threshold doubles as both reconnect-tolerance and orphan-reaping**. Splitting it into two independent timers resolves the tension between "allow reconnection" and "failproof termination":

| Timer | Renewed by | Window | Job | Why it's safe under spotty connections |
|---|---|---|---|---|
| **Tenant reconnect** | OBS **source present** (per tenant) | **3 min** | Allows reconnection. Source gone > 3 min → reap *that tenant* (destroy its GPU backend, stop its pipeline, detach). | Even a flapping OBS connection keeps resetting it; only true sustained absence reaps. |
| **Box lease** | the relay's **heartbeat to Vercel** | **~120 s** (≈ 12 missed 10 s beats) | Orphan catchall. Heartbeats stop → box is a zombie → destroy the whole box, all tenants. | Rides the **datacenter→Vercel** link, *not* the user's home uplink — user-side jitter cannot trip it. |

Plus a **per-tenant** absolute `hard_expires_at = created_at + 12 h` (confirm-to-extend, never auto-extended). The **shared box has no hard cap.**

### 9.2 The universal lease row

Every billable resource — *any* provider, *any* kind — is a row carrying the same four columns, so a single sweeper can reason about all of them:

```
(provider TEXT, provider_id TEXT, kind TEXT, renew_deadline TIMESTAMPTZ)
```

- `provider` + `provider_id` → `getProvider(kind, provider).destroy(provider_id)` (registry dispatch; no per-provider branch in the sweeper).
- `kind` (e.g. `gpu` / `vps`) → selects the registry and the destroy signature.
- `renew_deadline` → bumped on each heartbeat; the sweeper destroys anything past it.

~80% already exists: `provider`/`provider_id` are on every row; `gpu_instances.max_session_at` already is the per-tenant hard cap. The migration adds `renew_deadline` to the three tables and (for the hub) nothing else — emptiness is derived, not stored.

### 9.3 Multi-tenant emptiness is DERIVED, never stored

> **Hard rule:** delete `session_count`. Never store a refcount.

The shared hub's emptiness is a query against the same lease timestamps that drive everything else:

```sql
SELECT count(*) FROM gpu_instances
WHERE vps_hub_id = :hub AND renew_deadline > now();
```

Reconciled by construction on every evaluation — no stored number to desync. A lost detach can't strand it; a dead tenant drops out of the count automatically. **This is the actual root-cause fix.**

### 9.4 The one sweeper

A single provider-blind function, runnable from any context:

```
for row in (SELECT * FROM <all three tables>
            WHERE now() > renew_deadline
               OR now() > max_session_at            -- per-tenant hard cap
               OR (kind='vps' AND derived_empty_since < now() - empty_grace)):
    teardown(row)        -- existing idempotent teardownInstance / teardownHub,
                         -- dispatched via getProvider(row.kind, row.provider).destroy(row.provider_id)
```

- **Heartbeat-driven:** invoked opportunistically from every status/heartbeat handler (drop the `isPodAgent` gate). Any live box drives reaping of stale ones within ~1 beat.
- **Cron floor:** keep a low-frequency `/api/cron/reap` purely for the all-idle case (nothing live to drive the sweep).
- **Complement (kept):** the name-prefix / managed-by-label `listInstances` reconcile stays — a CASCADE-deleted row has no `renew_deadline` to read, so the lease alone cannot catch a row-less leak. The two are complementary, not redundant.

## 11. Acceptance criteria

The redesign is done when **all** hold:

1. **No orphan survives both timers.** A box whose relay stops heartbeating is destroyed within ~120 s by the box lease, regardless of provider or DB-row state, with no cron involved.
2. **A wedged refcount is impossible.** There is no stored counter; hub emptiness is derived from live leases. The §2 incident cannot recur.
3. **Reconnection works.** OBS can drop and reconnect within 3 min to the *same* box; no healthy stream is killed by control-plane jitter (the box lease rides the datacenter link).
4. **Provider-agnostic.** Adding a new VPS or GPU provider requires only registry registration + a managed-by label — **zero** new watchdog/reaper/self-destruct code. A grep for `'vast'` / `'hetzner'` string literals in the termination path returns nothing load-bearing.
5. **No silent backstop loss.** Losing the cron does not remove the only killer (heartbeat-driven sweep covers it); a leaked primary IP is swept.
6. **Tests green:** SQL asserts on throwaway Postgres (fresh replay + live schema), sweeper/lease `tsx` unit tests, `npx tsc --noEmit`.
