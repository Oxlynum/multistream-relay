================================================================================
VASTBROKER — Why SRT pod spin-up is slow/failing on Vast, and how to fix it
================================================================================
Written 2026-06-26. Author: debugging session with Claude.

--------------------------------------------------------------------------------
TL;DR
--------------------------------------------------------------------------------
- TARGET: a stream must go live in UNDER 60 seconds, ideally under 30.
- TODAY: provisioning can run 300-400+ seconds and then FAIL with no stream.
- The SRT media pipeline itself is HEALTHY and fast. It is not the problem.
- The problem is the GPU BROKER (web/lib/gpu-broker.ts + app/api/gpu/provision).
  It was designed for RunPod (fast, curated, predictable) and is now pointed at
  Vast.ai (a slow-to-map, variable-quality marketplace). Vast violates the
  broker's core timing and reliability assumptions.
- The broker cascades through candidate pods SYNCHRONOUSLY inside a single
  300-second Vercel request. On Vast each attempt costs ~60-180s, so a few bad
  hosts blow past the 300s ceiling. Vercel then KILLS the function mid-cascade:
  logs are lost, a pod is orphaned, the dock gives up, srt_port never saves, and
  OBS never gets a URL to publish to.

--------------------------------------------------------------------------------
WHAT IS *NOT* THE PROBLEM (verified live, 2026-06-26)
--------------------------------------------------------------------------------
The SRT ingest path is proven end-to-end and is fast:
- Pushed a real HEVC/SRT stream from a laptop, over the public internet, through
  Vast's UDP port forwarding, into the pod's MediaMTX. It connected, authenticated
  with the passphrase, routed by streamid, and fired the runOnReady hook.
- The transcoder (relay/supervisor.py) is clean and unchanged by the SRT switch.
- MediaMTX, the loopback, the encoder-start hook: all good.

CONCLUSION: once a pod is actually up and OBS has its URL, streaming works. The
failure is entirely UPSTREAM, in getting a healthy pod provisioned and handing
its SRT URL to OBS quickly.

--------------------------------------------------------------------------------
THE ARCHITECTURE OF THE FAILURE
--------------------------------------------------------------------------------
A single "Go Live" crosses three codebases on three machines:

  OBS plugin (C++, your Mac)
    -> POST /api/gpu/provision  (TypeScript, Vercel)   <-- THE BOTTLENECK
         broker rents a Vast pod, then SYNCHRONOUSLY waits for it to boot and
         pass readiness probes; on failure it destroys it and tries the next,
         ALL inside one HTTP request capped at maxDuration = 300s.
            -> pod boots, agent pairs back, starts MediaMTX (Python, on the pod)
    -> plugin polls /api/gpu/status for the saved srt_url, then tells OBS to publish

The broker loop (gpu-broker.ts -> provisionGpu):
  for each ranked candidate (up to MAX_BOOT_ATTEMPTS = 5):
     create pod
     waitForIp()      -- poll Vast until IP + RTMP port + SRT port are mapped
     probeRtmp()      -- RTMP handshake to confirm MediaMTX is serving
     probeUdp()       -- advisory UDP echo
     if all pass -> return success (provision route then saves srt_port)
     else -> destroy pod, try next candidate

This "create -> wait -> test -> fail -> retry, serially, in one request" shape is
fine for RunPod. It is fundamentally wrong for Vast.

--------------------------------------------------------------------------------
THE TIMING MATH (this is the core issue)
--------------------------------------------------------------------------------
Hard ceiling: provision route maxDuration = 300s (web/app/api/gpu/provision).
Broker knobs: MAX_BOOT_ATTEMPTS = 5, READINESS_TIMEOUT_MS = 180_000 (per pod).

Cost of ONE attempt:

                          RunPod (designed for)     Vast (running on)
  Ports ready after boot  ~immediate (datacenter)   ~57s (UDP maps late; the
                                                     container even bounces
                                                     running->loading->running)
  Host quality            curated, boots reliably    random marketplace machines;
                                                     MANY fail the GPU self-test
  A FAILED attempt costs  ~15-20s                    ~60s typical, up to 180s if
                                                     the pod's container exits and
                                                     waitForIp() spins to timeout

Multiply by 5 boot attempts:
  RunPod:  5 x ~18s   = ~90s   -> fits inside 300s. Cascade works.
  Vast:    5 x ~60-95s = 300-475s -> EXCEEDS 300s. Vercel kills the function.

So on Vast, a run of bad/slow hosts cannot complete inside the budget. Note the
budget was never the problem on RunPod; Vast's slowness is what overruns it.

--------------------------------------------------------------------------------
WHY VAST PRODUCES SO MANY FAILED ATTEMPTS (the GPU self-test ordering)
--------------------------------------------------------------------------------
relay/agent.py boots in this order:
   1. pair with Vercel        (fails -> exit)
   2. GPU self-test (NVENC+NVDEC)  (fails -> exit BEFORE MediaMTX starts)
   3. start MediaMTX

Vast is a marketplace of individual machines. Some are GPU-blind
(CUDA_ERROR_NO_DEVICE) or hit the NVENC-in-container driver regression. On those
hosts the self-test FAILS, so the agent exits at step 2 and MediaMTX never runs.
The broker's probeRtmp then finds nothing, declares the pod bad, destroys it, and
cascades to the next candidate. Each such bad host burns a full slow attempt.

RunPod's curated datacenters almost never failed boot, so cascades were rare.
On Vast they are common, which is exactly when the 300s overrun bites.

--------------------------------------------------------------------------------
WHAT THE USER ACTUALLY EXPERIENCES (observed in production logs, 2026-06-26)
--------------------------------------------------------------------------------
A real Go Live attempt showed:
  - 3 pods pairing ~21s apart   (the broker cascading through hosts)
  - a DELETE /api/gpu            (the dock giving up and orphan-destroying)
  - repeated 401 Unauthorized pairs (a leftover orphaned pod whose key was revoked)
  - NO [broker]/[provision] logs at all (the long function was killed before it
    could flush its buffered logs -- which is *why* this was so hard to diagnose)
  - the gpu_instances row never got srt_port saved -> OBS never got a URL -> no
    stream, and the pod was torn down.

Every one of those symptoms is explained by "the cascade overran 300s and the
Vercel function was killed mid-flight."

--------------------------------------------------------------------------------
ROOT CAUSES (in priority order)
--------------------------------------------------------------------------------
1. SYNCHRONOUS CASCADE IN A TIME-CAPPED REQUEST.
   Trying N pods one-at-a-time inside a single 300s HTTP request cannot absorb
   Vast's slow, variable boot/port-mapping. This is the #1 structural mismatch.

2. SLOW PORT MAPPING ON VAST (~57s, sometimes with a container bounce).
   The readiness gate must wait for the SRT UDP port before it can hand OBS a URL.
   On cold hosts this is dominated by the image pull + Vast finalizing port
   forwards. (On a warm host with the image cached it can be ~15-20s -- see fixes.)

3. MARKETPLACE HOSTS FAIL THE GPU SELF-TEST, forcing frequent cascades, each of
   which costs a full slow attempt. More retries needed, each more expensive.

4. (Self-inflicted, 2026-06-26) The RTMP probe was changed to retry 3x before
   giving up, adding ~28s per failed attempt. This made the 300s overrun WORSE
   and should be reverted/reshaped.

--------------------------------------------------------------------------------
THE FIX DIRECTION (target: live in <30-60s)
--------------------------------------------------------------------------------
The transcoder is fine. Do NOT rebuild it. Redesign the BROKER + provision flow.
Concretely, in rough priority:

A. DECOUPLE PROVISIONING FROM THE REQUEST (kills the 300s guillotine).
   - POST /api/gpu/provision should kick off provisioning and return immediately.
   - Readiness + cascade run in the background (a queue/worker, or driven by the
     dock's existing /api/gpu/status poll). No single request holds the whole
     cascade, so there is no hard time ceiling to overrun, and logs survive.

B. RACE A FEW PODS IN PARALLEL, not serially (matches a marketplace).
   - Rent 2-3 of the best candidates AT ONCE. First one to pass readiness wins;
     immediately destroy the losers. Turns "5 x 60s serial" into "~60s total" and
     makes a bad host harmless instead of a 60-180s tax on the critical path.

C. REJECT BAD HOSTS FAST (don't pay ~60s to discover a dead pod).
   - The agent already self-terminates on a failed GPU self-test. Have it POST a
     failure signal to the cloud first, so the broker abandons that host in ~1s
     instead of waiting out waitForIp/probeRtmp.
   - Tighten waitForIp's handling of exited/errored Vast states so a dead container
     is detected immediately, not at the 180s timeout.
   - Maintain/automate MACHINE_DENYLIST from observed self-test failures so we stop
     re-renting known-bad machines.

D. ATTACK THE ~57s PORT-MAPPING DIRECTLY.
   - Prefer hosts that already have the relay image cached (Vast exposes this);
     a warm host boots far faster than one pulling ~0.23GB cold.
   - Keep the image tiny (it already is) and pinned so it stays in host caches.
   - Consider a WARM POOL: keep 1-2 pre-booted, pre-tested pods ready so a Go Live
     attaches to an existing pod in seconds instead of renting cold. (Costs a little
     idle GPU; buys sub-10s starts. Worth it if fast start is a product requirement.)

E. ONLY GATE ON WHAT OBS NEEDS.
   - OBS needs the SRT port (8890/udp) + a serving MediaMTX. The 8889/udp probe
     port is advisory and should never block or delay readiness. (Partially done
     2026-06-26.) Hand OBS the URL the instant 8890 + RTMP are good.

Realistic outcome:
   - B + C alone (parallel race + fast bad-host rejection) should bring a cold
     start to roughly the single-host boot time (~30-60s) reliably, instead of
     serial-cascade roulette.
   - D's warm pool is what gets it consistently under ~15-30s.
   - A (decoupling) is what makes it ROBUST -- it can never again silently overrun
     and kill itself mid-cascade.

================================================================================
OPTION 2: NUKE THE BROKER AND START OVER — a Vast-native design
================================================================================
If the cleanup in "THE FIX DIRECTION" feels like polishing a model that was never
shaped for Vast, here is what a from-scratch broker looks like. The goal is the
same: live in <60s, ideally <30s, and NEVER silently overrun. The difference is
that every assumption is Vast-native from the start.

--------------------------------------------------------------------------------
THE ONE BIG FLIP: pods PUSH readiness; the cloud stops PROBING
--------------------------------------------------------------------------------
Today the cloud (a serverless function) reaches OUT to each pod and probes it:
an RTMP handshake, a UDP echo, polling Vast's API for mapped ports. This is slow
and, from serverless, unreliable (UDP can't round-trip back to an ephemeral Vercel
socket -- which is why the UDP echo is already only "advisory").

Invert it. The POD is the only thing that actually knows when it is healthy:
it just pairs, runs its GPU self-test, and starts MediaMTX. So let the pod REPORT
its own readiness instead of the cloud guessing from outside:

   pod boots -> pair -> GPU self-test
      if self-test FAILS: POST /api/agent/failed {machine_id, reason}  (cloud
                          denylists the machine, abandons this pod in ~1s)
      if self-test PASSES: start MediaMTX, then
                          POST /api/agent/ready {ip, srt_port, ...}     (cloud
                          marks this pod READY; its SRT URL is now serveable)

This single change removes probeRtmp, probeUdp, and most of waitForIp from the
critical path. No external probing, no serverless-UDP flakiness, and bad hosts are
known in ~1 second instead of ~60.

(Addressing note -- CONFIRMED on a live pod 2026-06-26: Vast injects the mapped
public ports into the container as environment variables, available immediately at
boot, matching Vast's API exactly:
     PUBLIC_IPADDR        = 66.96.91.67     (public IP)
     VAST_UDP_PORT_8890   = 39921           (the SRT ingest host port)
     VAST_UDP_PORT_8889   = 30837           (probe)
     VAST_TCP_PORT_1935   = 33635           (RTMP beacon)
     VAST_CONTAINERLABEL  = C.42746164      (the instance id)
 So the pod builds its OWN complete URL with zero cloud round-trips:
     srt://${PUBLIC_IPADDR}:${VAST_UDP_PORT_8890}?streamid=publish:<key>&...
 The cloud never polls Vast for ports and never probes the pod. This was the one
 open unknown for the push-readiness design -- it is resolved. Caveat: the env var
 is the INTENDED mapping, set at container start; the external forwarding goes live
 within a few seconds of boot, so OBS's normal SRT connect-retry covers the gap.
 Bonus: the pod also has /root/.vast_api_key, so it can even self-destruct via the
 Vast API directly if we ever want pod-driven teardown.)

--------------------------------------------------------------------------------
THE SECOND FLIP: RACE in parallel, don't CASCADE in series
--------------------------------------------------------------------------------
Vast is a marketplace -- renting is cheap and disposable. So rent several at once
and let them compete, instead of trying one, waiting a minute, and trying the next.

   provision(session):
     pick the top N candidates (N = 2 or 3) by tier -> distance -> price
     create all N in parallel, each tagged with this session_id
     each pod self-reports ready/failed (see flip #1)
     FIRST pod to report ready WINS:
        - atomic compare-and-set on the session row (first writer wins the claim)
        - its SRT URL is saved to the session
        - every other pod for this session is told "you lost, self-destruct"
     if ALL N fail -> rent the next N (still cheap, still parallel)

Wall-clock becomes "time for the FASTEST of N hosts to boot" (~the single-host
floor) instead of "sum of several slow serial failures." One bad host is now
harmless -- it just isn't the winner -- instead of a 60-180s tax on every stream.

--------------------------------------------------------------------------------
THE THIRD FLIP: async + a state machine, not one long request
--------------------------------------------------------------------------------
Nothing holds the whole provision inside a single timed HTTP request. Provisioning
is a STATE on the session row that advances via fast, individual webhooks:

   session.state:
     REQUESTED  -> POST /api/gpu/provision creates the row, kicks off the race,
                   returns immediately (no waiting)
     RACING     -> N pods booting; each POSTs ready/failed as it resolves
     READY      -> a winner claimed it; srt_url is set; losers self-destruct
     STREAMING  -> OBS connected (heartbeat says streaming=true)
     ENDED      -> stop/idle/credits -> teardown

The OBS dock already polls /api/gpu/status every few seconds -- it just watches the
session advance to READY and grabs the URL. No 300s guillotine, so it can never
again be killed mid-flight and lose its logs. Provisioning can take as long as it
honestly needs (it won't, but it's no longer fighting a deadline).

--------------------------------------------------------------------------------
THE SUB-30s LEVER: a small WARM POOL
--------------------------------------------------------------------------------
Vast's cold-boot floor (~57s: image pull + port mapping) cannot be beaten by any
amount of cleverness on the cold path. The only way under it is to NOT be cold.

Keep K (1-2) pods pre-booted, self-tested, MediaMTX running, ports mapped, idle.
On Go Live, atomically CLAIM a warm pod and hand OBS its URL in seconds. Replenish
the pool in the background.

   - Go Live time becomes "claim a warm pod" = a few seconds, sub-10s realistic.
   - Cost: K idle GPUs (~$0.10-0.20/hr each). Tune K to demand; run the pool only
     during active hours, or scale it from recent traffic.
   - The race design above is still the COLD fallback when the pool is empty.

A warm pool is the single biggest win for the <30s target and is much easier to
build on top of the clean push/race/async model than bolted onto today's broker.

--------------------------------------------------------------------------------
WHAT YOU KEEP vs THROW AWAY
--------------------------------------------------------------------------------
KEEP (these are good and Vast-agnostic):
  - relay/ entirely: the transcoder (supervisor.py), MediaMTX config, the agent's
    pair + self-test + MediaMTX boot. Add two tiny POSTs (ready/failed).
  - lib/providers/vast.ts offer search + create() + the UDP -p port flags.
  - The ranking idea (tier -> distance -> price) and the all-in price ceiling.
  - The teardown primitive (pod-teardown.ts) and the orphan reaper as a backstop.
  - The pod-side safety watchdogs (idle, missed-heartbeat, max-session).

THROW AWAY / REPLACE:
  - The synchronous cascade loop in gpu-broker.ts (waitForIp + probeRtmp +
    probeUdp + the serial for-loop). This is the RunPod-shaped core.
  - The "all in one 300s request" provision shape.
  - Cloud-side probing of pods entirely.

NEW PIECES (small):
  - app/api/agent/ready  + app/api/agent/failed   (pod self-reports)
  - a session state machine on gpu_instances (or a new sessions table)
  - a parallel "rent N, first-ready-wins, kill losers" coordinator
  - optional: a warm-pool manager + a cheap cron/worker to keep it topped up

--------------------------------------------------------------------------------
EFFORT / RISK
--------------------------------------------------------------------------------
  - This is a focused rewrite of ONE subsystem (broker + provision route + two new
    pod endpoints), not the product. The transcoder, billing, OBS plugin UI, auth,
    and platform plumbing are untouched.
  - Risk is contained: you can build the new path behind a flag and A/B it against
    the old one on real Go Lives before cutting over.
  - Biggest unknown to de-risk FIRST: confirm how a pod learns its own Vast-mapped
    public ports (env vs one getStatus call). Test on a live pod before committing
    to the push-readiness design; everything else follows from that answer.
  - Rough order to build: (1) pod ready/failed endpoints + agent POSTs, (2) parallel
    race + first-ready-wins, (3) async state machine + dock reads it, (4) warm pool.
    Steps 1-2 alone should already get cold starts to ~30-60s reliably.

--------------------------------------------------------------------------------
FILE MAP (where this logic lives)
--------------------------------------------------------------------------------
  web/lib/gpu-broker.ts            the cascade loop, waitForIp/probeRtmp/probeUdp
  web/app/api/gpu/provision/route.ts  the 300s request that runs the cascade
  web/lib/datacenters.ts           READINESS_TIMEOUT_MS, MAX_BOOT_ATTEMPTS, etc.
  web/lib/providers/vast.ts        Vast offer search, create(), port mapping flags
  web/lib/providers/index.ts       ACTIVE_PROVIDERS (Vast only today)
  relay/agent.py                   pod boot order: pair -> self-test -> MediaMTX
  relay/mediamtx.yml               the SRT server (this part is healthy)
  slimcast-obs/src/relay-dock.cpp  Go Live trigger, status poll, orphan-destroy

================================================================================
Bottom line: the broker's "rent one, wait, fail, retry -- all in one capped
request" model is RunPod-shaped. Vast needs "race a few in parallel, reject bad
hosts in ~1s, and never hold the whole cascade in a single timed request." Fixing
that -- not the transcoder -- is what gets spin-up under a minute.
